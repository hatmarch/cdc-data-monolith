Demo Walkthrough

Introduction to Legacy Application

  1. Show the coolstore without the inventory. You can get the URL of the coolstore UI by running the following command

    COOLSTORE_URL=http://$(oc get route coolstore-ui -o jsonpath='{.spec.host}' -n cdc-coolstore)

  2. Explain that the inventory comes from another system. A legacy system.

  3. Navigate to the legacy system. You can get the URL to the legacy application by running the following command in the demo shell:

    LEGACY_URL=http://$(oc get route www -o jsonpath='{.spec.host}' -n ${PROJECT_PREFIX}-dev)

  4. Show that the inventory database is currently empty by clicking on the "Inventory" navbar

  5. Show that the legacy system accepts an upload of inventory data by clicking on the Home navbar element

  6. Next click on Choose file button which should open up a window like so:

  7. In the directory shown ($DEMO_HOME/example) choose the `coolstore-inventory.csv'

  8. Click Load File

  9. You should now see a screen of all the imported inventory.

    Keep this tab open as we will revisit this page later

  10. Reopen the $COOLSTORE_URL and notice that inventory has not been updated still

Kafka: Producers, Consumers, and Groups

In this demo we want to show the notion of producers and consumers and ideally topics without any of the k8 overlays

See here for some commands that can be used.

For this we setup 4 terminals.

  1. All four get setup in the same way. Run the following command first in each:

    # from the root of this repo
docker run -it -v ~/.kube:/home/jboss/.kube -v ~/.oh-my-zsh:/home/jboss/.oh-my-zsh -v $(pwd):/workspaces/cdc-data-monolith -w /workspaces/cdc-data-monolith quay.io/mhildenb/cdc-demo-shell /bin/zsh

  2. Once in the docker shell, ensure you are logged in (one you are logged into one you should be logged into all)

    oc whoami
    opentlc-mgr

  3. Next run the following command to setup your environment

    . ${DEMO_HOME}/example/scripts/producer-consumer-setup.sh

    • Terminal 1 (Producer)

    • Terminal 2

    • Terminal 3

    • Terminal 4

    Terminal one is for producing information on a topic. Run the following command to get started

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-console-producer.sh --bootstrap-server localhost:9092 --topic ${TOPIC}
    The terminal is ready to take input on the topic when you see the chevron (>)

    Terminal 2 will be our first consumer in the group consumer-group-1

    Run the following command to get it started:

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic ${TOPIC} --group consumer-group-1

    Terminal 3 will be our second consumer in the group consumer-group-1

    Run the following command to get it started:

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic ${TOPIC} --group consumer-group-1

    Terminal 4 will be our first consumer in the group consumer-group-solo

    Run the following command to get it started:

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic ${TOPIC} --group consumer-group-solo

  4. Enter the following

    • Terminal 1 (Producer)

    Test Event 1

  5. You should notice output in one of Terminal 2 or 3 and Terminal 4

  6. Enter the following

    • Terminal 1 (Producer)

    Test Event 2

  7. You should notice output in the same of Terminal 2 and 3 that received the event before and Terminal 4

  8. Next kill the consumer in either Terminal 2 or 3, whichever one got the previous two events, with ctrl+c

  9. Enter the following

    • Terminal 1 (Producer)

    Test Event 3

  10. This time the other terminal of consumer-group-1 should get the message as should Terminal 4

  11. Now kill the consumer in Terminal 4 with ctrl+c

  12. Enter the following:

    • Terminal 1 (Producer)

    Test Event 4

  13. You should only see the event appear in the remaining connected terminal

  14. Finally, let’s see how the consumer groups have gone. Let’s use the kafka-consumer-groups command to see:

    • Terminal 4

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group consumer-group-solo
oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group consumer-group-1

  15. Output should be similar to:

    Consumer group 'consumer-group-solo' has no active members.

GROUP               TOPIC                    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG             CONSUMER-ID     HOST            CLIENT-ID
consumer-group-solo test-producer-consumer   0          16              17              1               -               -               -
    GROUP            TOPIC                    PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG             CONSUMER-ID                                                      HOST            CLIENT-ID
consumer-group-1 test-producer-consumer   0          17              17              0               consumer-consumer-group-1-1-c2d8588c-996e-47df-bb46-66d948d4a9ea /10.128.2.16    consumer-consumer-group-1-1

  16. Next, let’s reset the offset of the only consumer group that is currently offline (has no active members)

    • Terminal 4

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --reset-offsets --to-earliest --group consumer-group-solo --topic ${TOPIC} --execute
    GROUP                          TOPIC                          PARTITION  NEW-OFFSET
consumer-group-solo            test-producer-consumer-1       0          0

  17. Finally, let’s restart a consumer of the consumer-group-solo

    • Terminal 4

    oc exec -it demo-kafka-0 -n ${KAFKA_PROJECT} -- bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic ${TOPIC} --group consumer-group-solo

  18. After a few seconds you should see the following output

    • Terminal 4

    Test Event 1
Test Event 2
Test Event 3
Test Event 4
Test Event 5

  19. Which shows that our offset has been reset to the start causing us to reread the events
